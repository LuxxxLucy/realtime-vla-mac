Real-time VLA on MacBook

Jialin Lu · luxxxlucy.github.io

April 2026 · in progress

Target

Run π0 (and π0.5) on Mac in realtime rates. Dexmal showed that we can run π0 with 30 Hz on a RTX 4090 — see Running VLAs at Real-time Speed v1 and v2.
Right now my numbers are based on my quite lame M4 MacBook Air. RTX 4090 is a much more powerful card.

Current Status

Worst case: If M4 cannot get there, we will fallback to the dirty road: move to a higher-spec M5 Pro/Max chip. The addition of neural accelerator and much wider memory bandwidth shall get us there.

The major part of the computation, the Language model part requires  1224 GFLOP compute, divided by the combined GPU + ANE peak (3.23 + 5.7 = 8.93 TFLOPS) = 137 ms → ~7.3 Hz. This is also assuming a perfect split across engines at peak throughput, not physically realisable — just the hardware-ceiling reference.

Progress

π0 on MacBook decomposes into three stages per cycle:

  • Vision — SigLIP-So400m, one 224×224 image → 256 tokens.
  • Language — Gemma-1-2B prefix, bidirectional over 304 tokens (256 vision + 48 text). Text follows the Dexmal testing example.
  • AE (action expert) — Gemma-300M, 10 Euler steps of flow matching over 51 suffix tokens.

The PyTorch-MPS baseline spends 140 / 697 / 606 ms on V / L / AE — L dominates, and inside L the MLPs are 92 % of the FLOPs.

Phase 0 — fork the bottleneck apart. MLX port of L (fused Metal SDPA + steel_gemm_fused), alternative Metal backends (ggml / llama.cpp), hand-written fused-MLP kernels, ANE-GPU MLP bridging, and token-parallel MLP all characterise the ceilings; realtime-vla-mac-phase0 has the logs.

Phase 1 — ship it. Move the dispatch to pure Objective-C + CoreML, measure every stage on every engine, spread V, L, AE across three distinct engines so the steady-state cycle becomes max(V, L, AE) instead of sum(…).

Every stage lives on its own engine — L on ANE doesn't contend with V on GPU doesn't contend with AE on CPU. Cycle is bottlenecked by L on ANE under concurrent load. V and AE slow ~20 % from standalone because they share Unified Memory bandwidth; L on ANE stays flat because ANE has a dedicated datapath. Main open risks: sustained-cycle thermal behaviour (fanless M4 Air), fp16 Euler drift in AE (fp32 accumulator fix pending), and whether K=1 semantics of π0's 2.5 s action chunk cap the reactive rate.

Correctness: V / L match PyTorch fp32 with cos ≥ 0.9999 on every engine (ANE / GPU / CPU); AE matches to cos ≥ 0.99 (limited by fp16 Euler accumulator, engine-independent). CPU routing verified via asmonitor — ANE stays idle while MLComputeUnits.cpuOnly is in effect.
