Real-time VLA on MacBook

Jialin Lu · luxxxlucy.github.io

April 2026 · in progress

Target

Run π0 (and π0.5) on Mac in realtime rates. Dexmal showed that we can run π0 with 30 Hz on a RTX 4090 — see Running VLAs at Real-time Speed v1 and v2.
Right now my numbers are based on my quite lame M4 MacBook Air. RTX 4090 is a much more powerful card.

Current Status

Worst case: If M4 cannot get there, we will fallback to the dirty road: move to a higher-spec M5 Pro/Max chip. The addition of neural accelerator and much wider memory bandwidth shall get us there.

The major part of the computation, the Language model part requires  224 GFLOP compute, divided by the combined GPU + ANE peak (3.23 + 5.7 = 8.93 TFLOPS) = 137 ms → ~7.3 Hz. This is also assuming a perfect split across engines at peak throughput and hides the other compute of, not physically realizable, just the hardware-ceiling reference.

Progress

π0 on MacBook decomposes into three stages per cycle:

  • Vision — SigLIP-So400m, one 224×224 image → 256 tokens.
  • Language — Gemma-1-2B prefix, bidirectional over 304 tokens (256 vision + 48 text). The text is following the Dexmal testing example
  • AE (action expert) — Gemma-300M, 10 Euler steps of flow matching over 51 suffix tokens.

The PyTorch-MPS baseline spends 140 / 697 / 606 ms on V / L / AE — L dominates, and inside L the MLPs are 92 % of the FLOPs.

L: 697 ms → 227 ms. Four steps got here:

  1. MLX port of the prefix (fused Metal SDPA + steel_gemm_fused): 697 → 440 ms. Hand-written Metal kernels and ggml/llama.cpp did not beat it.
  2. MLP on ANE via CoreML, attention on GPU via MLX: 440 → 385 ms. ANE hits 14.5 ms/layer on a single-layer MLP; 18-layer chained is 18.4 ms/layer, not so ideal.
  3. Token-parallel MLP. For each MLP, which is now the bulk of bottleneck, we split [0:a] tokens to GPU, [a:N] to ANE concurrently; grid search lands at a=104 to GPU and rest to ANE has the best result: now we have 385 → 331 ms.
  4. Swift + CoreML + GCD dispatch, removing Python glue: 331 → 227 ms. This is current status.

Pipeline change: we can have async AE ‖ Vnext. Once L finishes, the current frame's AE and the next frame's V are independent and can run on different engines. GCD parallelism of them should give us (AE → ANE, Vnext → GPU) around 82 ms/cycle.

First-action is 3.6× the PyTorch-MPS baseline (1 237 → 348 ms). 30 Hz is hardware-limited on base M4 — the 224 ms L floor alone is 8× the RTX 4090's 27 ms cycle. Next levers: port V and AE to ANE CoreML, measure the MLX → MPS KV bridge for AE, then engine-parallel runtime. Main open risks: AE → ANE conversion (coremltools op gaps, dynamic KV), MLX ↔ MPS bridge cost on 17 MB KV, and whether K=1 semantics of π0's 2.5 s action chunk caps the reactive rate.
